##### 規模感

- チームメンバー
  - 正社員: 4 人（自分含む）
  - 業務委託: 3 人

##### 担当した役割

- 仕様理解・作成
  - PHP の設計書が残ってないためソースコードを正として新たに仕様書を書きおこした
    - 不明点・疑問点は都度、ツールを利用する企画・営業部門に相談・確認
- バックエンド（API）・バッチ・フロント(Thymeleaf)
  - 基本設計
    - 事前に書き起こした仕様をもとに、入力値・返却値を設計
  - 詳細設計
    - 基本設計を実現するためのデータフロー図・シーケンス図の作成
  - 実装
    - DDD（ドメイン駆動設計）を意識した実装
    - 早期リターンの意識
    - 基本的に一度代入した変数は変更しない
    - UT を意識して 1 メソッドに複数の役割を持たせない
  - UT
    - カバレッジ 100%を目指す（実際にコンフィグクラスを省き 100%近い数値を達成）
    - 条件網羅が原則
  - IT
    - 事前に書き起こした基本設計からテストケースの作成・実施
    - 例外処理も含めて実施

##### 困難だった箇所

- PHP の理解・仕様把握
  - 入社直後かつ PHP の経験がそれほどない中で、ドメイン知識を必要とする社内ツールの仕様把握は苦しかった
    - とりわけドメイン知識の習得が難しく、メンバーの理解者には何度も確認・相談を行なった
  - PHP がレポジトリの分散・継承・変数の書き換えの多用・ネストが深い・多重配列の多用と処理を追うのが大変だった
    - 一つ一つ丁寧にメモをとり、特に複雑に思える部分では、メンバーとダブルチェックを行った
- 実装
  - 自分の担当となった API に実装が複雑なものがあった
    - 施設 A がある企業 B の傘下に入った際に施設 A の契約を本契約から企業 B に紐づく子契約に変更するもの
    - 一度ダミー施設を作成して、そのダミー契約の一部値を施設 A の契約と入れ替える処理があり非常に複雑
    - PHP の処理読み込みから担当したが、メンバーと何度も相談をして仕様を把握した
    - 社内の別プラットフォームに API 経由で問い合わせする処理があり、その PF に対する知識を理解するのがまた大変だった

##### 成果

- PHP ⇨ Java へのリプレイスが完了
  - レポジトリが 2 つになったことで管理が楽に
  - DDD を意識してドメイン操作の処理が 1 パッケージにまとめられたことでクラスの役割が明確になった
    - クラスの役割が明確になったことで統一した実装がしやすい
  - API とフロントに切り分けたことで障害時に原因の究明がしやすくなった
    - フロントが API からエラーを受け取っているのか、フロントがエラーを起こしているのか切り分けが容易になった
